The problem

Documentation doesn't
keep pace with code

  • A component is refactored — the docs are never updated
  • A new developer copies a stale example into production
  • Prop names change; the docs still show the old names
  • AI tools hallucinate because they're given outdated context
  • Team time is wasted answering "how does X work?" in Slack
"The codebase has strong patterns — but those patterns only exist in the heads of senior developers."

Every convention — @brdp/ui, defineMessages, FormFields<T> — is invisible to anyone who wasn't there when it was decided.

This tool makes those conventions explicit, enforced, and auto-documented.

Trade-off analysis — advantages

Why this approach works

Speed

Zero-friction generation

One command. A full .md or .mdx file in seconds — lives right next to your source file, versioned in git like everything else.

Consistency

Conventions encoded, not assumed

The prompt template carries your rules. Every output already follows team standards — without anyone having to remember them.

Bidirectional

Docs → Code too

The to-ts command lets you scaffold a component from its markdown spec. Design the API first, generate the implementation second.

Onboarding

Living knowledge base

New team members get accurate, up-to-date docs for every component they touch. Senior developers stop answering the same questions.

LLM Context

Better AI assistance

Generated .md files become accurate context for Copilot, Claude, or any AI tool. Better input → better output.

Scalable

Works across the monorepo

Runs from any directory. Can be scripted into Turborepo pipelines to regenerate docs automatically on merge.

Trade-off analysis — limitations

Where it falls short

≠
Output is non-deterministic Two runs of the same file may produce slightly different docs. Review is still required.
?
Business context is invisible The AI can't know this component is used in the multi-sig flow unless that's written in the code itself.
⟳
Garbage in, garbage out Poorly typed or poorly structured code produces poorly described docs. Quality starts at the source.
$
API cost per call Fine for on-demand use. Becomes a factor if wired into CI for every commit.
∿
Can be over-verbose Simple utilities may get more documentation than they need. Manual trimming is sometimes faster.
↻
Templates need maintenance As team conventions evolve, the prompt files must evolve too. They're not fire-and-forget.
The right mental model

Think: first draft, not final copy

The AI writes the 80% that is structural and predictable. You add the business context, the gotchas, and the non-obvious edge cases.

A 70% accurate first draft is still faster than starting from a blank file.

Three commands, each wired to a prompt template:

Command Template Flow
to-md ts-to-md.txt .tsx → .md
to-mdx ts-to-mdx.txt .tsx → .mdx
to-ts md-to-ts.txt .md → .tsx

Retry logic: up to 2 retries with exponential backoff. Only NetworkError is retried — ApiError surfaces immediately.

Stdout mode: omit the [output] argument and the result is piped to stdout — composable with other shell tools.

Prompt engineering

How the templates work

Each .txt file in /prompts is a parameterised prompt. The CLI injects your source via {'{CODE}'} or {'{DOCS}'}: 

// PromptUtils.inject() replaces {VARS}
const prompt = PromptUtils.inject(template, {
  [PromptVar.Code]: sourceCode
});

// Result: "...read the code below:\n\n" + your file

The template instructs the model on:

  • The codebase context — @brdp/ui, defineMessages, Jalali dates
  • The required output structure — exact sections, order, table columns
  • The team rules — enforced automatically in every output
  • The analysis step — forces the model to reason before writing

Structure of ts-to-md.txt:

①

System role + codebase context

"You are a technical doc writer for an internal banking UI built on @brdp/ui…"

②

Analysis step (silent reasoning)

Forces classification: component / hook / utility / form / feature

③

Required output structure

Exact sections in exact order. Skipped sections must include <!-- N/A -->

④

Hard rules section

Validator signature, monetary values, type only, fallback operators

⑤

{'{CODE}'} injection point

Your source file is pasted here at runtime

Three ways to use it

API

Inject as context

Read the .md at runtime and pass it as the system prompt. The model knows your component's exact API, props, and constraints before answering. 

const CONFIG: AppConfig = {
  API: {
    HOSTNAME: process.env["API_HOSTNAME"],
    MODEL:    process.env["API_MODEL"],
    KEY:      process.env["API_KEY"],
    TIMEOUT:  60_000,
  },
  PATHS: {
    PACKAGE_ROOT,
    PROMPTS: path.join(PACKAGE_ROOT, "prompts"),
  },
} as const;
✓ Full control over model, retries, output
✓ Composable into any pipeline or script
✗ Needs API key + code to wire up
Chat

Paste & generate

Open Claude.ai or ChatGPT. Paste the .md and ask it to implement. Accurate doc = accurate code, no hallucinated props or wrong patterns.

Claude.ai ChatGPT Gemini
✓ Zero setup, works right now
✓ Good for one-off or exploratory use
✗ Manual copy-paste, not repeatable
✗ Context window fills up on large docs
Editor tools

Give your IDE the context

Drop the .md in your project. Attach it directly or let the tool pick it up — generate, autocomplete, and refactor with full awareness of your API.

Cursor attach doc in composer
Claude Code reads project docs natively
Continue.dev @folder to load full context
Copilot open doc tab for implicit context
✓ Stays in your normal dev flow
✓ Autocomplete aware of your exact props

What the prompt enforces

Your team conventions,
baked in

Every generated doc and component already follows these rules — because they live in the prompt template:

Rule Why it matters
type only, never interface Team standard
Import from @brdp/ui, never antd Abstraction layer
All strings via defineMessages + getFormatMessage() i18n enforcement
FormFields<T> annotation required Type safety
Validator: (rule, value) — no third param Our validator shape
?? not || for fallbacks Null-safety
Monetary values: string → addCommaToAmount() No coercion bugs
Use dependencies + hidden/disabled callbacks No useEffect for field logic

Example: instead of useEffect watching a field, the generated code uses dependencies: 

// ✗ what the AI will NOT generate
const type = fieldWatch('paymentType');
useEffect(() => {
  setFieldHidden('iban', type !== 'transfer');
}, [type]);

// ✓ what the AI generates instead
{
  name: 'iban',
  hidden: ({ paymentType }) => paymentType !== 'transfer',
  dependencies: ['paymentType'],
}

Adding a convention is a one-line change in the prompt file. Update it once — every future doc generation reflects it immediately.

What comes next

Roadmap

Short term — prompts

Ant Design–style prompt structure

Update prompt templates to produce output matching the exact antd format — consistent API tables, usage examples, and design token sections in every generated doc.

Short term — conventions

Living prompt versioning

Track prompt templates in git like source code. Every convention change is a PR — reviewable, diffable, and rolled back if it produces worse output. No more silent prompt drift.

Medium term — inference

Fully local AI

All inference on-prem. No cloud calls, no data leaving the network. One config line to swap models. 

model: "deepseek-coder:6.7b"
backend: "ollama"
api_url: "http://localhost:11434"
Medium term — pipeline

CI / Turborepo integration

Docs regenerate automatically on merge — never out of sync with source.

  • Turborepo task: docs:generate
  • Re-runs only when .tsx files change
  • PR bot comments on changed doc sections
Long term — agents

Multi-agent orchestration

Inspired by this Divar engineering post — specialised agents routed by an orchestrator, each owning one job. Outputs merged and conflicts resolved automatically.

Long term — changelog

Live changelog generation

Git diff → structured entries with emoji, PR links, and contributor mentions.
